Testing Type

Unit Test

A unit test is a piece of a code (usually a method) that invokes another piece of code and checks the correctness of some assumptions afterward. If the assumptions turn out to be wrong, the unit test has failed.

-- The Art of Unit Testing

Runs in process

Can be run at a click of a button or a single command

The Good

  • Fast
  • Precise Location of Failure

The Bad

  • Require Test Doubles (Mocking)
  • Limits Refactoring

Require Practice

Testing Type

Integration Test

Is a Unit Test

Except: Can execute code out of process.

The Good

  • Simulates Real Usage
  • Improved Refactoring Options

The Bad

  • Slow
  • Problem of State
  • Failure may Require Debugging

Easier to learn

compared to Unit Tests

Testing Type

Behavioral Test

Tests driven by the use cases / requirements.

The Good

  • Closest to the Business Outcome
  • Easily Communicates Progress
  • Business Users can Write Tests
  • Great Open Source Tools

The Bad

  • VERY Slow
  • Problem of State (x 2)
  • Failure WILL Require Debugging
  • Requires Sophisticated Business User

Requires Custom Fixtures

Testing Type

User Interface Test

Tests that simulate the user.

Includes system tests.

The Good

  • Best Chance of Catching Errors

The Bad

  • EXTREMELY Slow
  • Does not Work for All Test Cases
  • Expensive to Maintain: Brittle

Requires "Safe Testing" Area

Testing Discipline

  • Test First
  • Test Driven Development
  • Test Last

Test First

  1. One or More Test Cases are Written
  2. Enough Code is Written to Make Tests Pass
  3. Refactor

Test Driven Development (TDD)

Red, Green, Refactor

  1. Create a unit tests that fails.
  2. Write production code that makes that test pass.
  3. Clean up the mess you just made.

Test Last

Tests are written after the production code is written.

Test Last == Test Never

Testing with Current Date / Time

public interface IClock
{
	DateTime GetCurrentDate();
}

public class RealClock : IClock
{
	public DateTime GetCurrentDate()
	{
		return DateTime.Now;
	}
}

Testing with Current Date / Time

public class ClockForTesting : IClock
{
	private readonly DateTime _currentDateTime;

	public ClockForTesting(
		DateTime currentDateTime)
	{
		_currentDateTime = currentDateTime;
	}

	public DateTime GetCurrentDate()
	{
		return _currentDateTime;
	}
}

Also Testable

public DateTime GetFirstDayOfMonth(IClock clock)

	var currentDay = clock.GetCurrentDate();
